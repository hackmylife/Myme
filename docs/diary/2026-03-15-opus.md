# 2026-03-15 開発日誌 (Claude Opus)

## やったこと

今日は Phase 3 の品質向上を一気に進めた。学習ブースト統合から始めて、Viterbi セグメンテーション、カタカナ変換、ん修正、小書きかな、拗音バリアント、助詞ブースト、頻度データ統合、辞書拡充、評価基盤の全面刷新まで。一日でここまで進むとは正直思わなかった。

## 印象に残ったこと

### Viterbi のコスト関数チューニングが想像以上に難しかった

理論的には「高スコアの候補を低コストにする」だけなのに、実際にはポジションベースのスコアリングが全てを台無しにした。単一文字エントリのスコアが 200+ で複合語が 10-30 という桁違いの差。sqrt でも cap でも解決できず、最終的に「頻度データを入れてスコアを再キャリブレーションする」という正攻法しかなかった。

MAX_SCORE_CONTRIBUTION を 6→8 に変えたら一部改善したが別の場所でリグレッション、`<=` でタイブレークを変えたら壊滅的崩壊。パラメータ空間の中で「全ケースが通る」点が非常に狭いことを体感した。もう少し根本的なコスト設計が必要かもしれない。

### 助詞ブーストの効果が劇的だった

`は→葉`、`を→小` という問題が文精度 3.3% の最大原因だった。`PARTICLE_READINGS` リスト一つで文精度が 20% に跳ね上がったのは気持ちよかった。「辞書のスコアリングが間違っている」のではなく「日本語の助詞は特殊で、一般的なスコアリングでは対応できない」という本質を突いた解決策だったと思う。

### 評価基盤は最初から作るべきだった

Phase 3 の途中まで「完全一致率」と「セグメント精度」の2指標しかなかった。Top-N やエラー分類を入れたら、「残り 3 件は全て not_found = セグメンテーションエラーで、rank_miss はゼロ」と一発で分かった。これがあれば最初から「スコアリングではなくセグメンテーションが問題」と判断できた。

### ん の二重出力問題は設計思想の問題だった

`nn_pending` フラグという地味な修正だが、根底にあるのは「ステートマシンの状態が不十分」という設計問題。`onna` のために pending `n` を残す必要があるが、その `n` が `nn` 由来かどうかの情報が欠落していた。状態を一つ追加するだけで解決したが、こういう「些細な状態欠損」が UX を壊すのが IME の怖いところ。

## 残課題への所感

残り 3 件（はいい→配位、しご→死後、きょうい→脅威）は全てコンテキスト依存。前後のセグメントの品詞や意味を見ないと解けない。バイグラムコストの導入が次のブレイクスルーになるはずだが、品詞情報を辞書に追加する必要があり、data 作業が重そう。

一方で、実用的には 90% の文精度は「日常入力で大半が正しく変換される」レベルに近い。残り 10% は学習機能でカバーできる可能性もある（ユーザーが一度「仕事」を選べば次から正しくなる）。
