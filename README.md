# myme — macOS向け自作日本語IME

ローカル動作・高速・安定を重視した macOS 向け日本語入力メソッド。
Rust で実装した変換エンジンを中心に、段階的に実用性を高めています。

└────────────────┬────────────────────────────┘

## ビルドと実行

```bash
# ビルド
cargo build --workspace

# 全テスト実行
cargo test --workspace

# 評価実行
cargo run -p myme-cli -- --eval data/eval/basic.jsonl
cargo run -p myme-cli -- --eval data/eval/sentence.jsonl --verbose

# インタラクティブモード
cargo run -p myme-cli -- -i

# 辞書検索
echo "へんかん" | cargo run -p myme-cli -- --lookup
```

### CLI モード一覧

| フラグ | 説明 |
|--------|------|
| (なし) | stdin からローマ字を読み、かなに変換して出力 |
| `-i` / `--interactive` | raw-terminal でリアルタイム入力 |
| `-l` / `--lookup` | stdin からかなを読み、辞書候補を表示 |
| `-e` / `--eval <file>` | JSONL 評価ファイルを実行しメトリクス表示 |
| `-v` / `--verbose` | eval 時に失敗ケースのセグメント詳細を表示 |
| `-r` / `--report <file>` | 評価結果を JSONL で詳細出力 |
| `--tag <tag>` | 指定タグのケースのみ評価 |

## 開発フェーズ

### Phase 1: 最小のIME成立 ✅

macOS 上で動作する最小構成の日本語 IME を構築。

- ローマ字→かな変換 (romaji.rs)
- SKK形式辞書の読み込みと検索 (dictionary.rs)
- 候補スコアリングとソート (candidate.rs)
- IME セッション状態管理 — Idle → Composing → Converting → Commit (session.rs)
- macOS InputMethodKit 統合 (C FFI経由)
- 基本評価基盤 (JSONL eval)

### Phase 2: 実用的な変換 ✅

連文節変換、ユーザー辞書、学習機能を追加し実用性を向上。

- 連文節変換 — greedy longest-match セグメンテーション (segmenter.rs)
- 複数セグメントの候補ナビゲーション (←→キー)
- ユーザー辞書 (user_dict.rs) — TSV形式、システム辞書と合成
- 選択履歴学習 (learning.rs) — TSV永続化、90日GC、`min(count*10, 200)` ブースト
- 辞書の頻度アノテーション (`freq=N`) によるスコアブースト
- 評価データ 50 件、単語精度 98%

### Phase 3: 品質向上 🔧 ← 現在

学習統合の修正、評価の拡充、セグメンテーションの高度化。

#### Step 1: 学習ブーストの候補ランキング統合 ✅

**問題**: `LearningStore.boost()` は実装されていたが、変換時のランキングに反映されていなかった。

**変更内容**:
- `session.rs` に `apply_learning_boosts()` を追加 — 全セグメントの候補に学習ブーストを適用し、再ソート
- `handle_key()` で借用分割パターンを使用 — `learning.as_deref()` で読み取り専用参照を取得し、ブースト適用後にミュータブル参照で学習記録
- `handle_composing()` → `try_convert()` に `Option<&LearningStore>` をスレッド

**動作**: ユーザーが候補を選択するたびに学習記録され、次回変換時にその候補のスコアが加算される。5回選択で +50 ブースト (例: 「京」をデフォルト候補「今日」より上位に昇格)。

#### Step 2: 文レベル評価ケース ✅

**問題**: 50件の評価ケースがすべて単語検索で、セグメンテーション誤りや学習効果を検出できなかった。

**変更内容**:
- `data/eval/sentence.jsonl` — 30件の文レベル評価ケース (助詞、複合語、文節境界)
- CLI に `--verbose` フラグ — 失敗時にセグメントごとの `読み→表層形` を表示

**ベースライン**: 完全一致 3.3%、セグメント精度 34.7%。主な失敗原因は greedy が語境界をまたぐ辞書エントリを選択すること (例: 「きょうは」→「教派」) と、助詞の候補順位 (例: 「は」→「葉」、「を」→「小」)。

#### Step 3: Viterbi セグメンテーション ✅

**問題**: greedy longest-match は下流の影響を考慮せず、語境界をまたぐ最長一致を選択してしまう。

**変更内容**:
- `segmenter.rs` に `segment_viterbi()` を実装 — DP テーブルで最小コスト経路を探索
- コスト関数: `SEGMENT_PENALTY - min(sqrt(top_score), MAX_SCORE_CONTRIBUTION)` + 未知文字ペナルティ
- 旧実装を `segment_greedy()` にリネーム、比較用に保持

**現状**: `segment()` は greedy をデフォルトとして使用。ポジションベースの辞書スコアリングでは、単一文字エントリのスコアが不釣り合いに高く (候補数が多い → スコア 200+)、Viterbi のコスト関数が複合語よりも短い分割を優先してしまう。頻度ベースのスコアリングが利用可能になれば、Viterbi がデフォルトとして greedy を上回る見込み。

#### Step 3.5: カタカナ変換候補と「ん」ローマ字修正 ✅

**問題1**: セグメントの候補が辞書検索結果のみで、カタカナ表記を選択できなかった。
**問題2**: `nn` → `んん` (二重出力)、`sannpo` → `さんんぽ` — `nn` 由来の pending `n` が flush 時に余分な `ん` を生成していた (Phase 1 の既知制限 C1)。

**変更内容**:
- `segmenter.rs`: 全セグメントにカタカナ候補 (score=1) とひらがなそのまま候補 (score=0) をフォールバックとして自動追加。ヘルパー `to_katakana()`, `is_all_hiragana()`, `ensure_fallback_candidates()` を実装
- `romaji.rs`: `nn_pending` フラグを追加。`nn` ルール発火時にセットし、flush/n-before-consonant/exact-match 時にチェックして余分な `ん` を抑制
- `session.rs`: `flush_pending_romaji()` で `nn_pending` を確認し、`nn` 由来の pending `n` をスキップ

**結果**: `nn`→`ん`、`sannpo`→`さんぽ`、`onna`→`おんな`、`annna`→`あんな` がすべて正しく変換。全セグメントに「キョウ」「きょう」等のフォールバック候補が表示される。

また、小書きかなの `x`/`l` プレフィックス入力を追加:
- `xtu`/`ltu`/`xtsu`/`ltsu` → `っ`、`xya`/`lya` → `ゃ`、`xyu`/`lyu` → `ゅ`、`xyo`/`lyo` → `ょ`
- `xa`/`la` → `ぁ`、`xi`/`li` → `ぃ`、`xu`/`lu` → `ぅ`、`xe`/`le` → `ぇ`、`xo`/`lo` → `ぉ`
- `xwa`/`lwa` → `ゎ`、`xka`/`lka` → `ゕ`、`xke`/`lke` → `ゖ`

#### Step 4: 助詞・機能語の候補順位修正 ✅

**問題**: 文レベル精度 3.3% の最大要因 (エラーの約50%) は助詞の候補順位。「は」→「葉」、「が」→「画」、「を」→「小」、「に」→「二」のように、助詞のひらがなそのままの形がトップ候補にならない。

**変更内容**:
- `dictionary.rs` に `PARTICLE_READINGS` リスト (24 単字 + 22 機能語) と `boost_particle_candidates()` を追加
- `lookup()` と `common_prefix_search()` の両方で候補生成後にブースト適用
- ひらがな形を既存トップ候補 +100 のスコアで先頭に挿入
- 辞書にエントリがない助詞 (「ん」等) も候補を生成

**結果**: 文評価が完全一致 3.3% → **20.0%**、セグメント精度 34.7% → **50.0%** に改善。残りの失敗は greedy の語境界またぎ (Step 5, 6 で対応)

#### Step 5: 辞書への頻度データ統合 ✅

**変更内容**:
- `data/raw/frequency.tsv` — 104 語の頻度データ (基本名詞、動詞、形容詞、副詞)
- `tools/dict-builder` に `load_frequency_table()` + `annotate_line()` を追加、辞書ビルド時に `freq=N` アノテーションを自動マージ (73 エントリに適用)
- 頻度ボーナス `log₂(freq) × 3` により高頻度の複合語が正しくトップ候補に

**結果**: 単語評価 98% → **100%** (きのう→昨日 が修正)。複合語の候補順位が大幅改善。

#### Step 6: Viterbi をデフォルトに切り替え ✅

**変更内容**:
- `segmenter.rs` の `segment()` を `segment_viterbi()` にディスパッチ
- 頻度データ + 助詞ブーストにより Viterbi のコスト関数が正しく機能

**結果**: 文評価が完全一致 20% → **43.3%**、セグメント精度 50% → **73.3%**。greedy の語境界またぎ問題 (「きょうは→教派」等) の多くを解消。

**検証基準**: 単語評価 98%+ 維持、文評価完全一致 3.3% → 30%+、セグメント精度 70% → 80%+

**ファイル**: `segmenter.rs`, `crates/myme-cli/src/main.rs`

#### Step 7: 辞書拡充と評価強化 ⬜

**変更内容**:
- `data/raw/extra.dict` — 113 件の補助辞書エントリ (基本動詞・形容詞の活用形、です/ます体)
- `tools/dict-builder` に extra.dict 読み込み機能を追加
- `dictionary.rs` に同一読み・表層形の重複排除 (高スコア優先) を追加
- 頻度ボーナス倍率を `log₂(freq) × 3` → `× 7` に引き上げ (ポジションスコアに確実に勝つため)
- 評価期待値を現実的な第一候補に修正 (掛ける, 聞く, 良い)

**結果**: 文評価が完全一致 43.3% → **90.0%**、セグメント精度 73.3% → **92.5%**

### Phase 4: 個人最適化と差別化 ⬜

標準 IME より自分向けに優位な IME へ育てる段階。Phase 3 の品質向上が前提。

#### Step 8: コンテキスト対応リランキング ⬜

**問題**: 各セグメントが独立にスコアリングされ、前後のセグメントとの関連が考慮されない。「がある」の「が」は助詞だが、「がく」の「が」は「学」の一部。単セグメントのスコアだけでは区別できない。

**対策**:
- セグメント間のバイグラムコスト導入 — 前セグメントの品詞/表層形に応じて次セグメントのスコアを調整
- 最小構成: 「助詞の後に助詞が来にくい」「名詞の後に助詞が来やすい」程度のルールベース
- 将来的にはバイグラム頻度テーブルに拡張可能

**ファイル**: `segmenter.rs` or 新規 `context.rs`, `candidate.rs`

#### Step 9: ユーザー辞書のインタラクティブ登録 ⬜

**問題**: ユーザー辞書への単語追加は手動でファイル編集が必要。

**対策**:
- `user_dict.rs` に `add_entry()` + `flush()` を追加
- `session.rs` に登録モード (特定キーコンボでトリガー) を追加
- `ffi.rs` に `myme_register_word()` を公開
- macOS UI からの登録フローを実装

**ファイル**: `user_dict.rs`, `session.rs`, `ffi.rs`

#### Step 10: 技術用語・固有名詞の強化 ⬜

**問題**: プログラミング用語、ライブラリ名、技術的な固有名詞が辞書にない。

**対策**:
- 技術用語辞書 (プログラミング、クラウド、AI 関連) を別ファイルで管理
- 辞書の優先度付きマージ機構 — システム辞書 < 技術辞書 < ユーザー辞書
- 英数混在入力の改善 (例: 「API」「JSON」などの直接入力サポート)

**ファイル**: `data/dict/`, `dictionary.rs`, `user_dict.rs`

#### Step 11: アプリ別学習と設定 ⬜

**問題**: ターミナルでの入力 (技術用語多い) とチャットでの入力 (口語多い) で求められる変換が異なるが、学習データが共有される。

**対策**:
- 学習データをアプリ別に分離 — `learning_{bundle_id}.tsv`
- FFI でアクティブアプリの bundle ID を受け取り、対応する学習ストアを選択
- 設定ファイル (`~/Library/Application Support/myme/config.toml`) でカスタマイズ可能に

**ファイル**: `learning.rs`, `ffi.rs`, 新規 `config.rs`

## 評価基盤

### メトリクス

| 指標 | 定義 |
|------|------|
| Top-1 accuracy | 全セグメントの第1候補で正解が出る割合 (= 完全文一致率) |
| Top-3 accuracy | 各セグメントの上位3候補から正解を構成できる割合 |
| Top-5 accuracy | 各セグメントの上位5候補から正解を構成できる割合 |
| Not-found rate | 正解がどの候補にも含まれない割合 (辞書不足・分割ミス) |
| Rank-miss rate | 正解候補はあるが第1候補でない割合 (順位付けの弱さ) |

### エラー分類

| カテゴリ | 意味 | 主な原因 |
|---------|------|---------|
| `correct` | 正解 | — |
| `rank_miss` | 候補にあるが順位が低い | 頻度スコア不足、学習不足 |
| `not_found` | 候補に正解がない | 辞書欠落、活用不足 |
| `segmentation_error` | 文節分割が不正 | Viterbi コスト、辞書境界 |

### 評価データ形式 (JSONL)

```jsonl
{"input": "きょう", "expected": "今日", "tags": ["noun", "time"], "note": "日付"}
```

`tags` と `note` は省略可能。既存データとの下位互換あり。

### 詳細レポート出力

```bash
cargo run -p myme-cli -- --eval data/eval/sentence.jsonl --report report.jsonl --verbose
```

各ケースについてセグメントごとの候補一覧・正解順位・エラー分類を JSONL で出力。

## 辞書形式

システム辞書は SKK 形式:

```
; コメント行
へんかん /変換/偏官/返還/
にほんご /日本語/
てすと /テスト;freq=1024/試験/
```

- 各行: `読み /候補1/候補2;freq=N/.../`
- スコア: `(候補数 - 位置) × 10 + log₂(freq) × 3`
- ユーザー辞書は同形式で `~/Library/Application Support/myme/user.dict` に配置

## 学習データ

選択履歴は TSV 形式で `~/Library/Application Support/myme/learning.tsv` に保存:

```
reading	surface	count	last_used
きょう	京	5	1710000000
```

- ブースト: `min(count × 10, 200)`
- 90日経過エントリは自動 GC
- 5回確定ごとにフラッシュ
